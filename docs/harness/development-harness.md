# WWT Development Harness

## 1. 목표

WWT 클라이언트 개발의 기준선을 문서화해, 구현 속도보다 일관성과 품질을 먼저 확보한다.

## 2. 적용 범위

- 인증/회원
- 상품 조회/등록/수정/삭제
- 거래 요청
- 채팅
- 공통 에러 처리/상태 처리

## 3. 핵심 흐름

1. 회원가입(이메일 인증) -> 로그인
2. 상품 탐색/검색 -> 상세 조회
3. 거래 요청 -> 수락/거절
4. 채팅방 생성 -> 메시지 송수신

## 4. 기술 하네스 규칙

1. API Base URL: `/api`
2. 인증 헤더: `Authorization: Bearer {accessToken}`
3. 상태 모델 분리:
   - auth
   - products
   - tradeRequests
   - chats
4. 에러 처리 규칙:
   - 분기 키는 항상 `code`
   - 401은 refresh 시도 후 실패 시 로그아웃
   - 403/404/409는 사용자 행동 가능한 메시지로 안내
5. 페이지네이션 키(`page`, `limit`)는 모든 목록 화면에서 일관 적용

## 5. 구현 순서 (권장)

1. 인증 체인
2. 상품 조회 체인
3. 거래 요청 체인
4. 채팅 체인
5. 작성/수정/삭제 및 마이페이지

## 6. DoD (Definition of Done)

1. 명세 endpoint와 필드 매핑 일치
2. 성공/실패 상태 UI 존재
3. 명세 에러코드 분기 구현 완료
4. 테스트 시나리오 통과
5. 문서(결정 로그/테스트 시나리오) 업데이트 완료

## 7. 리스크 가드

- RFR 토큰 재사용 탐지 시 세션 즉시 만료
- 민감 데이터 저장 위치를 명시하고 Git 제외
- 삭제/권한 액션은 항상 서버 응답 기준으로 최종 확정

## 8. 산출물

- 기능 코드
- 테스트 코드
- 결정 로그
- 시나리오 문서
