## 시나리오 이름

- 목적: 앱 첫 진입 화면(`/`)이 요청된 리스트형 마켓 디자인으로 노출되는지 검증한다.
- 사전 조건:
  - 개발 서버 실행(`npm run dev`)
  - 홈 화면 접근 가능
- 입력 데이터:
  - 홈 샘플 리스트 아이템 데이터

## 절차

1. `/` 접속 후 상단에 지역 선택 버튼과 3개 액션 버튼(검색/필터/알림)이 보이는지 확인한다.
2. 상품 카드형 리스트가 세로로 노출되고 각 항목에 이미지/제목/메타/가격/반응 수치가 표시되는지 확인한다.
3. 우하단 플로팅 `+` 버튼과 하단 5개 탭 메뉴가 표시되는지 확인한다.
4. 첫 번째 상품 항목 선택 시 상품 상세(`/products/1`)로 이동되는지 확인한다.

## 기대 결과

- 홈이 소개 랜딩이 아닌 마켓 피드형 첫 화면으로 동작한다.
- 레이아웃이 모바일 기준 스크린샷과 유사한 정보 밀도/구조를 제공한다.
- 기존 상세 페이지 이동과 스모크 테스트가 유지된다.

## 에러 케이스

- 401:
  - 홈 화면은 비인증 접근이 가능하고, 인증 필요 동작에서만 기존 흐름이 유지되어야 한다.
- 403:
  - 권한 제한 리소스 접근 시 홈 레이아웃이 깨지지 않아야 한다.
- 404:
  - 상세 이동 대상이 없을 때 코드 기반 오류 처리 페이지로 안전하게 전환되어야 한다.
- 409:
  - 해당 없음(홈 정적 리스트 렌더링 범위)
- 5xx:
  - 추후 API 연동 시 에러 노출 영역을 홈 구조 내에 유지해야 한다.

## 시나리오 이름

- 목적: UIUX 2차 개선 후 핵심 플로우(탐색/상세/채팅/인증)의 사용성 및 시각 일관성을 검증한다.
- 사전 조건:
  - 개발 서버 실행(`npm run dev`)
  - 데스크톱(1280px 이상), 모바일(560px 이하) 뷰포트 확인 가능
- 입력 데이터:
  - 기본 mock 상품/채팅 데이터
  - 데모 로그인 계정

## 절차

1. `/` 접속 후 브랜드/핵심 가치/행동 버튼이 첫 화면에서 즉시 구분되는지 확인한다.
2. `/products` 접속 후 검색/정렬/상품 리스트의 정보 위계가 유지되는지 확인한다.
3. `/products/[id]` 접속 후 가격/판매자/주요 행동 버튼(거래 요청/채팅)이 한 화면에서 식별 가능한지 확인한다.
4. `/auth/login`, `/auth/signup`에서 입력 폼 가독성/포커스/버튼 상태(활성/비활성)를 확인한다.
5. `/chats`, `/chats/[id]`, `/mypage`에서 목록 스캔성과 본문 가독성이 이전 대비 개선되었는지 확인한다.
6. 데스크톱/모바일 모두에서 상단~하단 채움 정책(데스크톱 중앙 캔버스, 모바일 풀스크린)이 유지되는지 확인한다.

## 기대 결과

- 모든 주요 화면이 동일한 타이포/간격/컬러 시스템을 공유한다.
- 사용자가 첫 화면 3초 내에 서비스 정체성과 다음 행동을 이해할 수 있다.
- 폼/리스트/상세/채팅 화면에서 텍스트 대비와 클릭 가능한 요소가 명확하다.

## 에러 케이스

- 401:
  - 인증 만료 시 기존 `code` 분기 및 세션 정리 흐름이 유지되어야 한다.
- 403:
  - 권한 제한 리소스 접근 시 안내 문구가 레이아웃 깨짐 없이 표시되어야 한다.
- 404:
  - 잘못된 상품/채팅 경로에서 공통 오류 UI 가독성이 유지되어야 한다.
- 409:
  - 중복 요청 에러 노출 시 버튼/폼 상태가 중복 제출 없이 유지되어야 한다.
- 5xx:
  - 서버 오류 메시지 노출 시 주요 액션 버튼이 비정상 상태에 빠지지 않아야 한다.

## 시나리오 이름

- 목적: WWT 전용 UI/레이아웃/PWA가 데스크톱/모바일 요구사항에 맞게 동작하는지 검증한다.
- 사전 조건:
  - 개발 서버 실행(`npm run dev`)
  - HTTPS 또는 localhost 환경(서비스워커 등록 가능)
- 입력 데이터:
  - 기본 데모 계정(`bluefish@wwt.dev`)
  - 상품 목록 mock 응답

## 절차

1. 데스크톱 뷰(가로 1280px 이상)에서 `/` 접속 후 앱 컨테이너가 상단부터 하단까지 채워지고 좌우만 여백인지 확인한다.
2. 모바일 뷰(가로 560px 이하)에서 `/`, `/products` 접속 후 화면이 풀스크린으로 채워지고 라운드 프레임/외곽 여백이 제거되는지 확인한다.
3. 홈 화면 문구가 DSM 학생 전용 중고거래 서비스 소개 내용인지 확인한다.
4. 브라우저 개발자도구 Application > Manifest에서 앱 이름/색상/아이콘이 로드되는지 확인한다.
5. Application > Service Workers에서 `sw.js`가 등록되는지 확인한다.

## 기대 결과

- 화이트 배경 기반 UI와 포인트 컬러가 전역에 일관 적용된다.
- 데스크톱은 세로 전 높이 앱 캔버스 + 좌우 여백 레이아웃을 유지한다.
- 모바일은 화면 전체를 앱이 채운다.
- Manifest/Service Worker가 정상 적용되어 PWA 설치 가능 상태가 된다.

## 에러 케이스

- 401:
  - 로그인 만료 시 기존 refresh 분기(`code`)가 유지되어야 한다.
- 403:
  - 권한 없는 리소스 접근 시 코드 기반 안내가 유지되어야 한다.
- 404:
  - 잘못된 상품/채팅 경로 접근 시 코드 기반 안내가 유지되어야 한다.
- 409:
  - 거래 요청/회원가입 충돌 코드 노출 규칙이 깨지지 않아야 한다.
- 5xx:
  - 서버 장애 시 공통 에러 안내 UI가 유지되어야 한다.

## 시나리오 이름

- 목적: 초기 세팅 이후 기본 라우팅/조회 UI 및 테스트 하네스가 정상 동작하는지 검증한다.
- 사전 조건:
  - `npm install` 완료
  - `.env.local` 미설정 시 기본 API Base URL(`/api`) 사용
- 입력 데이터:
  - 로그인 입력값(유효/무효 이메일)
  - 상품 목록 API 응답(성공/실패)

## 절차

1. `npm run test`를 실행해 단위 테스트를 통과한다.
2. `npm run dev` 실행 후 `/`에서 `상품 목록` 링크 이동을 확인한다.
3. `/auth/login`에서 이메일/비밀번호 입력 후 로그인 요청을 시도한다.
4. `/products`에서 로딩/성공/실패 UI 중 하나가 정상 렌더링되는지 확인한다.
5. `npm run test:e2e`로 홈->상품목록 스모크 시나리오를 실행한다.

## 기대 결과

- Vitest 테스트가 모두 통과한다.
- Playwright 스모크 테스트가 통과한다.
- 로그인/상품 목록 화면에서 오류 발생 시 `code` 기준으로 에러 메시지가 노출된다.
- API 401 응답 시 refresh 시도 로직이 동작하고, refresh 실패 코드에 따라 세션이 정리된다.

## 에러 케이스

- 401:
  - `UNAUTHORIZED` 시 refresh 재시도
  - `INVALID_REFRESH_TOKEN`, `EXPIRED_REFRESH_TOKEN`, `REFRESH_TOKEN_REUSE_DETECTED` 시 세션 정리
- 403:
  - 공통 `FORBIDDEN` 메시지 노출
- 404:
  - `PRODUCT_NOT_FOUND`, `CHAT_ROOM_NOT_FOUND` 등 코드 기반 메시지 처리
- 409:
  - `DUPLICATE_PENDING_TRADE_REQUEST`, `SELF_TRADE_NOT_ALLOWED` 분기 기반 재시도 안내
- 5xx:
  - 공통 서버 장애 메시지 및 재시도 유도

## 시나리오 이름

- 목적: 회원가입(코드 발송/검증/가입)이 `order.md` 구현 요구와 API 명세 코드 체계에 맞게 동작하는지 검증한다.
- 사전 조건:
  - 개발 서버 실행(`npm run dev`)
  - 기존 가입 계정(`bluefish@wwt.dev`) 외 신규 이메일 사용
- 입력 데이터:
  - 이메일: `new-user@wwt.dev`
  - 인증코드: `123456` (mock 고정)
  - 닉네임: `newbie`
  - 비밀번호: `password123`

## 절차

1. `/auth/signup` 접속 후 이메일 입력, `인증코드 요청` 클릭.
2. 인증코드 `123456` 입력 후 `인증코드 검증` 클릭.
3. 닉네임/비밀번호 입력 후 `회원가입` 클릭.
4. `/auth/login`으로 리다이렉트되는지 확인.

## 기대 결과

- 1단계에서 `POST /api/auth/signup/email-code`가 `201`로 성공한다.
- 2단계에서 `POST /api/auth/signup/email-code/verify`가 `200`으로 토큰을 반환한다.
- 3단계에서 `POST /api/auth/signup`이 `201`로 유저를 생성한다.
- 실패 시 `message`가 아닌 `code` 기반 에러 분기가 표시된다.

## 에러 케이스

- 401:
  - 해당 없음(회원가입 체인)
- 403:
  - 해당 없음(회원가입 체인)
- 404:
  - 해당 없음(회원가입 체인)
- 409:
  - `EMAIL_ALREADY_EXISTS` 반환 및 에러 노출
- 5xx:
  - 공통 서버 장애 메시지 노출
